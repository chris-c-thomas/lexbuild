# CLAUDE.md — LexBuild Astro App

## Project Overview

`apps/astro/` is the next-generation LexBuild web application — an Astro-powered, multi-source legal content browser deployed to a self-managed VPS (AWS Lightsail) behind Cloudflare's edge cache. It replaces `apps/web/` (Next.js on Vercel) as the production frontend.

The app serves the full converted output of every LexBuild source package at every granularity level: U.S. Code (54 titles, ~60k sections) and eCFR (50 titles, ~200k+ sections), with architecture ready for additional corpora (annual CFR, Federal Register, state statutes).

Like `apps/web/`, this app has **no code dependency** on `@lexbuild/core`, `@lexbuild/usc`, or `@lexbuild/ecfr`. It consumes their output — `.md` files with YAML frontmatter and `_meta.json` sidecar files — as data.

### Key Architectural Differences from `apps/web/`

| Concern | `apps/web/` (Next.js) | `apps/astro/` (Astro) |
|---|---|---|
| Framework | Next.js 16, App Router, full RSC | Astro 5.x, island architecture |
| JS shipped to client | React hydrates entire page tree | Zero JS by default; React islands only where interactive |
| Runtime | Vercel Serverless Functions | Long-lived Node.js process (PM2) on VPS |
| Content storage | Vercel Blob / Cloudflare R2 via S3 SDK | Local filesystem (`fs.readFile`) |
| Search | Pagefind (client-side static index) | Meilisearch (server-side, proxied) — Phase 3 |
| Syntax highlighting | Shiki at request time (server) | Pre-rendered HTML files from build pipeline |
| CDN | Vercel edge cache (ISR) | Cloudflare proxy (orange-cloud, `Cache-Control` headers) |
| Sources | USC only | USC + eCFR (extensible to more) |
| Cost | ~$20+/mo (Vercel Pro + Blob) | ~$20/mo fixed (Lightsail) + $0 (Cloudflare free) |

## Architecture Summary

### SSR with Filesystem Content + CDN Edge Caching

Every page is server-rendered by Astro's Node adapter. The server reads `.md` files from the local filesystem, parses frontmatter, serves pre-rendered highlighted HTML (or falls back to runtime Shiki), renders the Markdown body to HTML, and returns a complete response. Cloudflare caches the response at the edge based on `Cache-Control` headers.

Request flow:
```
Browser → Cloudflare Edge (cache HIT? → serve) → Origin (Lightsail VPS)
          → Caddy (TLS termination, reverse proxy)
          → Astro Node server (port 4321)
          → fs.readFile() from /srv/lexbuild/content/
          → Response with Cache-Control: public, s-maxage=31536000, stale-while-revalidate=86400
```

After the first request for any URL, Cloudflare serves it from edge cache. Origin traffic is minimal.

### Local Development Flow

For local dev, the app reads from a `content/` directory inside the app (symlinked or copied from CLI output). No VPS, no Cloudflare, no external dependencies.

```
Browser → localhost:4321 (Astro dev server)
        → fs.readFile() from ./content/ (or CONTENT_DIR env var)
        → Runtime Shiki highlighting (if no .highlighted.html present)
```

### Island Architecture

Astro renders pages as static HTML with zero JavaScript by default. Interactive components are React islands hydrated only where needed:

| Component | Island? | Reason |
|---|---|---|
| `BaseLayout.astro` | No | Static HTML shell |
| `FrontmatterPanel.astro` | No | Pure display |
| `BreadcrumbNav.astro` | No | Pure display |
| `ContentViewer` | **Yes** (`client:load`) | Source/preview toggle, clipboard, download |
| `Sidebar` | **Yes** (`client:load`) | Accordion state, lazy JSON fetch, virtualization, active-item tracking |
| `SearchDialog` | **Yes** (`client:idle`) | Meilisearch client, keyboard shortcut (Phase 3) |
| `ThemeToggle` | **Yes** (`client:load`) | Dark mode toggle, localStorage |

## Content Directory Convention

The app expects a unified content directory matching the CLI's output structure. For local dev, this is achieved via symlinks or file copies from the monorepo `output/` directory.

### Local Dev Setup Script (`scripts/link-content.sh`)

```bash
#!/bin/bash
# Link CLI output into the app's content directory for local development.
# Run from apps/astro/ after running lexbuild convert-usc and convert-ecfr.
set -euo pipefail

REPO_ROOT="$(cd ../.. && pwd)"
OUTPUT_DIR="${REPO_ROOT}/output"
CONTENT_DIR="./content"

# Ensure output exists
if [ ! -d "$OUTPUT_DIR/usc" ] && [ ! -d "$OUTPUT_DIR/ecfr" ]; then
  echo "Error: No output found at $OUTPUT_DIR"
  echo "Run 'lexbuild convert-usc --all' and 'lexbuild convert-ecfr --all' first."
  exit 1
fi

# Create content directory structure
mkdir -p "$CONTENT_DIR"/{section,chapter,title}/{usc,ecfr}
mkdir -p "$CONTENT_DIR"/part/ecfr

# Symlink section-level output (the primary granularity for browsing)
# USC: output/usc/title-NN/chapter-NN/section-N.md → content/section/usc/title-NN/...
if [ -d "$OUTPUT_DIR/usc" ]; then
  rm -f "$CONTENT_DIR/section/usc"
  ln -s "$OUTPUT_DIR/usc" "$CONTENT_DIR/section/usc"
  echo "Linked USC section output"
fi

# eCFR: output/ecfr/title-NN/chapter-X/part-N/section-N.N.md → content/section/ecfr/title-NN/...
if [ -d "$OUTPUT_DIR/ecfr" ]; then
  rm -f "$CONTENT_DIR/section/ecfr"
  ln -s "$OUTPUT_DIR/ecfr" "$CONTENT_DIR/section/ecfr"
  echo "Linked eCFR section output"
fi

# Note: chapter-level and title-level output requires running the CLI with -g chapter / -g title.
# Those are optional for local dev — the app gracefully handles missing granularity levels.

echo "Content linked. Run 'pnpm dev' to start the dev server."
```

### Content File Resolution

The content reader resolves paths relative to `CONTENT_DIR`:

```
CONTENT_DIR/
├── section/usc/title-01/chapter-01/section-1.md
├── section/usc/title-01/chapter-01/section-1.highlighted.html  (optional)
├── section/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.md
├── section/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.highlighted.html  (optional)
├── chapter/usc/title-01/chapter-01/chapter-01.md
├── chapter/ecfr/title-17/chapter-IV/chapter-IV.md
├── part/ecfr/title-17/chapter-IV/part-240.md
├── title/usc/title-01.md
└── title/ecfr/title-17.md
```

**Important**: The default CLI output at section granularity writes to `output/usc/` and `output/ecfr/` — these ARE the section-level files. The symlink approach above maps `output/usc/` directly to `content/section/usc/`. For chapter, part, and title granularities, the CLI must be run separately with the appropriate `-g` flag and the output placed accordingly.

For local dev, having only section-level output is sufficient to browse individual sections. Chapter, part, and title views will 404 gracefully if those granularities haven't been generated.

## Route Architecture

### Source-Prefixed Catch-All Routes

Each source gets its own route tree under `pages/{source}/`. A catch-all `[...slug].astro` route handles all granularity levels within that source.

#### USC Routes (`pages/usc/[...slug].astro`)

| URL Pattern | Slug | Granularity | Content Path |
|---|---|---|---|
| `/usc/title-01` | `["title-01"]` | title | `title/usc/title-01.md` |
| `/usc/title-01/chapter-01` | `["title-01", "chapter-01"]` | chapter | `chapter/usc/title-01/chapter-01/chapter-01.md` |
| `/usc/title-01/chapter-01/section-1` | `["title-01", "chapter-01", "section-1"]` | section | `section/usc/title-01/chapter-01/section-1.md` |

#### eCFR Routes (`pages/ecfr/[...slug].astro`)

| URL Pattern | Slug | Granularity | Content Path |
|---|---|---|---|
| `/ecfr/title-17` | `["title-17"]` | title | `title/ecfr/title-17.md` |
| `/ecfr/title-17/chapter-IV` | `["title-17", "chapter-IV"]` | chapter | `chapter/ecfr/title-17/chapter-IV/chapter-IV.md` |
| `/ecfr/title-17/chapter-IV/part-240` | `["title-17", "chapter-IV", "part-240"]` | part | `part/ecfr/title-17/chapter-IV/part-240.md` |
| `/ecfr/title-17/chapter-IV/part-240/section-240.10b-5` | `["title-17", "chapter-IV", "part-240", "section-240.10b-5"]` | section | `section/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.md` |

### Slug Resolution Logic (`src/lib/routes.ts`)

The catch-all route receives `Astro.params.slug` as a string array. Resolution rules:

```typescript
/**
 * Resolve a slug array to a content file path and granularity.
 *
 * USC:
 *   1 segment  → title     (title-NN)
 *   2 segments → chapter   (title-NN/chapter-NN)
 *   3 segments → section   (title-NN/chapter-NN/section-N)
 *
 * eCFR:
 *   1 segment  → title     (title-NN)
 *   2 segments → chapter   (title-NN/chapter-X)
 *   3 segments → part      (title-NN/chapter-X/part-N)
 *   4 segments → section   (title-NN/chapter-X/part-N/section-N.N)
 */
```

Key differences between sources:
- USC has 3 levels (title/chapter/section). eCFR has 4 (title/chapter/part/section).
- USC chapter numbers are zero-padded Arabic (chapter-01). eCFR chapter numbers are Roman (chapter-IV).
- eCFR section numbers are dot-delimited (section-240.10b-5). USC section numbers can be alphanumeric (section-7801, section-202a).
- Both sources use string section numbers — never parse as integers.

### Slug Validation

Every slug segment is validated before filesystem access:
1. Must match expected prefix pattern (`title-`, `chapter-`, `part-`, `section-`)
2. Must not contain `..`, `/`, `\`, or null bytes (path traversal prevention)
3. Must resolve to a file that exists on disk

Invalid slugs return 404 via `Astro.response.status = 404`.

## Core Library Modules

### `src/lib/content.ts` — Filesystem Content Reader

```typescript
import { readFile, access } from "node:fs/promises";
import { resolve, relative } from "node:path";

const CONTENT_ROOT = resolve(process.env.CONTENT_DIR ?? "./content");

/** Validate path stays within CONTENT_ROOT. Prevents traversal. */
function safePath(subpath: string): string {
  const resolved = resolve(CONTENT_ROOT, subpath);
  const rel = relative(CONTENT_ROOT, resolved);
  if (rel.startsWith("..") || rel.includes("\0")) {
    throw new Error(`Path traversal blocked: ${subpath}`);
  }
  return resolved;
}

/** Read a content file. Returns null if not found. */
export async function getFile(path: string): Promise<string | null> {
  try {
    return await readFile(safePath(path), "utf-8");
  } catch {
    return null;
  }
}

/** Check if a content file exists. */
export async function fileExists(path: string): Promise<boolean> {
  try {
    await access(safePath(path));
    return true;
  } catch {
    return false;
  }
}
```

No provider abstraction, no factory pattern, no dynamic imports. The app reads from the filesystem — period. On the VPS, content lives at `/srv/lexbuild/content/`. Locally, it's `./content/` (symlinked from `output/`).

### `src/lib/highlight.ts` — Pre-Rendered Highlight Loader

```typescript
import { getFile } from "./content";

/**
 * Load pre-rendered syntax-highlighted HTML for a content file.
 * Falls back to runtime Shiki if .highlighted.html doesn't exist.
 */
export async function getHighlightedHtml(
  contentPath: string,
  rawMarkdown: string,
): Promise<string> {
  // Try pre-rendered file first (produced by build pipeline)
  const highlightPath = contentPath.replace(/\.md$/, ".highlighted.html");
  const preRendered = await getFile(highlightPath);
  if (preRendered) return preRendered;

  // Fallback: runtime Shiki (local dev without pre-rendered highlights)
  const { highlightMarkdown } = await import("./shiki");
  return highlightMarkdown(rawMarkdown);
}
```

### `src/lib/shiki.ts` — Runtime Shiki Singleton (Local Dev Fallback)

Same pattern as `apps/web/`. Module-level singleton persists across requests in the long-lived Node process:

```typescript
import type { BundledLanguage, BundledTheme, HighlighterGeneric } from "shiki";
import { createHighlighter } from "shiki";

let highlighterPromise: Promise<HighlighterGeneric<BundledLanguage, BundledTheme>> | null = null;

function getHighlighter() {
  if (!highlighterPromise) {
    highlighterPromise = createHighlighter({
      themes: ["github-light-default", "github-dark-default"],
      langs: ["markdown", "yaml"],
    });
  }
  return highlighterPromise;
}

/** Highlight Markdown with dual themes for light/dark mode. */
export async function highlightMarkdown(code: string): Promise<string> {
  const hl = await getHighlighter();
  return hl.codeToHtml(code, {
    lang: "markdown",
    themes: { light: "github-light-default", dark: "github-dark-default" },
  });
}
```

**Note**: On the VPS, `.highlighted.html` files will always be present (generated during content pipeline). Runtime Shiki is only used during local development when you haven't generated highlight files. This keeps the production runtime lightweight.

```typescript
const sanitizeSchema = {
  ...defaultSchema,
  attributes: {
    ...defaultSchema.attributes,
    "*": [...(defaultSchema.attributes?.["*"] ?? []), "className", "id"],
  },
};

const processor = unified()
  .use(remarkParse)
  .use(remarkGfm)
  .use(remarkRehype, { allowDangerousHtml: false })
  .use(rehypeSanitize, sanitizeSchema)
  .use(rehypeSlug)
  .use(rehypeStringify);

/** Render Markdown body (without frontmatter) to sanitized HTML. */
export async function renderMarkdown(markdown: string): Promise<string> {
  const result = await processor.process(markdown);
  return String(result);
}
```

### `src/lib/routes.ts` — Slug Resolution

```typescript
import type { SourceId, Granularity, ResolvedRoute, Breadcrumb } from "./types";
import { getSource, SOURCES } from "./sources";

/**
 * Resolve a source + slug array into a content path and metadata.
 * Returns null if the slug doesn't match expected patterns.
 */
export function resolveRoute(
  sourceId: SourceId,
  slug: string[] | undefined,
): ResolvedRoute | null {
  if (!slug || slug.length === 0) return null;

  const source = getSource(sourceId);
  const granularity = source.slugGranularity[slug.length];
  if (!granularity) return null;

  // Validate each segment against expected prefix
  for (const segment of slug) {
    if (!isValidSegment(segment)) return null;
  }

  const segments = parseSegments(sourceId, slug);
  const contentPath = buildContentPath(sourceId, granularity, slug);
  const highlightPath = contentPath.replace(/\.md$/, ".highlighted.html");
  const breadcrumbs = buildBreadcrumbs(sourceId, slug);
  const canonicalUrl = `${source.basePath}/${slug.join("/")}`;

  return {
    source: sourceId,
    granularity,
    contentPath,
    highlightPath,
    segments,
    breadcrumbs,
    canonicalUrl,
  };
}

function isValidSegment(segment: string): boolean {
  // Allow: title-NN, chapter-NN, chapter-IV, part-NNN, section-NNN.NNN, section-NNNa
  // Block: .., /, \, null bytes
  return (
    /^(title|chapter|part|section)-[\w.-]+$/.test(segment) &&
    !segment.includes("..") &&
    !segment.includes("\0")
  );
}

function buildContentPath(
  sourceId: SourceId,
  granularity: Granularity,
  slug: string[],
): string {
  // granularity/ + sourceId/ + slug path + filename
  switch (granularity) {
    case "title":
      return `title/${sourceId}/${slug[0]}.md`;
    case "chapter": {
      const last = slug[slug.length - 1];
      return `chapter/${sourceId}/${slug.join("/")}/${last}.md`;
    }
    case "part": {
      const last = slug[slug.length - 1];
      return `part/${sourceId}/${slug.join("/")}/${last}.md`;
    }
    case "section":
      return `section/${sourceId}/${slug.join("/")}/${slug[slug.length - 1]}.md`;
    // Note: eCFR section path differs — section file sits inside the part directory.
    // The actual path is: section/ecfr/title-NN/chapter-X/part-N/section-N.N.md
    // which matches slug.join("/") naturally since the slug includes all segments.
  }
}

// buildBreadcrumbs and parseSegments implementations follow similar patterns.
```

**Important edge case for eCFR section paths**: The CLI writes eCFR sections into part directories as `output/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.md`. When symlinked to `content/section/ecfr/`, the full path is `content/section/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.md`. The slug `["title-17", "chapter-IV", "part-240", "section-240.10b-5"]` joined is `title-17/chapter-IV/part-240/section-240.10b-5`, which resolves correctly to `section/ecfr/title-17/chapter-IV/part-240/section-240.10b-5.md`.

However, the section `.md` file itself lives directly in the part directory (not in a `section-N.N/` subdirectory). So `buildContentPath` for eCFR sections should be:
```
section/ecfr/{slug[0]}/{slug[1]}/{slug[2]}/{slug[3]}.md
```
NOT:
```
section/ecfr/{slug[0]}/{slug[1]}/{slug[2]}/{slug[3]}/{slug[3]}.md  ← WRONG
```

This differs from USC sections where the file is `section-N.md` inside a chapter directory. Test both source paths during implementation.

```astro
import type { SourceId } from "../../lib/types";

const SOURCE_ID: SourceId = "usc";

// Resolve route

if (!route) return Astro.redirect("/404");

// Read content

// Get highlighted source HTML (pre-rendered or runtime fallback)

// Render Markdown body to HTML
const renderedHtml = await renderMarkdown(body);

// Build page title
const pageTitle = frontmatter.title ?? `${SOURCE_ID.toUpperCase()} - ${slug}`;

// Cache headers — CDN caches for 1 year, revalidates in background
Astro.response.headers.set(
  "Cache-Control",

  <BreadcrumbNav breadcrumbs={route.breadcrumbs} />

    rawMarkdown={body}
```

## Shiki Pre-Rendering Strategy

### Build Pipeline Addition

After converting content with the CLI, run a highlighting step that generates `.highlighted.html` companion files. This is a new script, not a CLI flag — it runs independently of the conversion pipeline.

**`scripts/generate-highlights.ts`** (runs in the monorepo or on the VPS):

```typescript
/**
 * Generate pre-rendered Shiki HTML for all .md files in the content directory.
 * Writes a .highlighted.html file alongside each .md file.
 *
 * Usage: npx tsx scripts/generate-highlights.ts [content-dir]
 *
 * Performance: ~60k files in ~2-5 minutes (Shiki is fast for Markdown).
 * Can be parallelized with worker threads if needed.
 */
```

Approach:
1. Walk the content directory recursively
2. For each `.md` file, parse frontmatter with gray-matter, extract body
3. Highlight body with Shiki (dual themes: `github-light-default`, `github-dark-default`)
4. Write `{filename}.highlighted.html` alongside the `.md` file
5. Skip files that already have an up-to-date `.highlighted.html` (compare mtime)

This script can live in `apps/astro/scripts/` or at the monorepo root. It's independent of the Astro build.

### Local Dev Without Pre-Rendering

During local development, you can skip generating highlight files entirely. The app's `getHighlightedHtml()` function falls back to runtime Shiki when `.highlighted.html` is missing. This is slower on first request but acceptable for dev.

## Common Pitfalls

- **Astro `[...slug]` returns a string, not an array.** `Astro.params.slug` is `"title-01/chapter-01/section-1"` — split on `/` to get segments.
- **Section numbers are strings.** `"202a"`, `"7701-1"`, `"240.10b-5"` are all valid. Never parse as integers.
- **eCFR chapter numbers are Roman numerals.** `chapter-I`, `chapter-IV`, not `chapter-01`. The sidebar, routes, and content paths must all preserve the original format.
- **eCFR has 4 hierarchy levels, USC has 3.** The slug resolver must handle both. A 3-segment slug means "section" for USC but "part" for eCFR.
- **Title-level files can be very large.** Title 26 (USC) is ~10MB. Title 17 (eCFR) is similarly large. Pre-rendered highlights avoid runtime Shiki cost. If runtime Shiki is used (local dev), expect 200ms+ for these files.
- **`content/` is git-ignored.** It must be populated before the dev server works. Run `scripts/link-content.sh` after CLI conversion.
- **`public/nav/` is git-ignored.** Run `scripts/generate-nav.ts` before the sidebar works. Without nav JSON, the sidebar renders empty (no crash).
- **Tailwind CSS v4 requires `@tailwindcss/postcss` and a `postcss.config.mjs`.** Same pitfall as `apps/web/`.
- **Do NOT add this app to the default `build` task in turbo.json.** Only `build:astro` should build it.
- **Caddy handles TLS in production.** The Astro server listens on HTTP localhost only. Do not configure Astro to serve HTTPS.
- **`rehype-sanitize` is critical.** Content is user-visible legal text, but defense-in-depth against any injection in Markdown content.
- **PM2 restart is not zero-downtime by default.** Use `pm2 reload` (not `pm2 restart`) for graceful reload in production.
